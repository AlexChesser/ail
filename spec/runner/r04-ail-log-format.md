# r04. AIL Log Format Specification

## Purpose

This document defines the **ail-log/1** output format — the human-readable, terminal-safe representation of a completed pipeline run. It is the contract for:

- **Format implementors**: writers of formatters that convert stored pipeline events to ail-log
- **Parser authors**: anyone writing a grammar, syntax highlighter, or structured parser for ail-log documents
- **Extension authors**: consumers that render ail-log in an IDE or TUI

The ail-log format is the single external interface to run data. It is generated by `ail log [run_id]` and `ail log --follow` subcommands.

---

## 4.1 Version Header and File Format

### Version Header (Mandatory)

Every ail-log document **must** begin with the version header on its own line:

```
ail-log/1
```

This MUST be the first line of every output, whether streamed (`--follow`) or complete (`ail log`).

### File Extension

Files containing ail-log format use the extension `.ail-log`.

### Zero HTML Rule

**Critical:** The ail-log format MUST contain **no HTML, CSS, or Rich Text Markup**. All formatting must be:

- **Terminal-readable** — runnable through `cat`, `less`, `grep`, piped to other Unix tools
- **Greppable** — plaintext directives and Markdown only
- **Plain UTF-8** — printable ASCII + Unicode for symbols only

This ensures:
- Backward compatibility with line-based text tools
- Streaming compatibility (incremental reads per line)
- Accessibility (no JavaScript, no rendering engine required)
- Portability (any terminal, any platform)

---

## 4.2 Directive Syntax

Directives are **structured blocks** that contain metadata or non-Markdown content (thinking, tool calls, results, subprocess output).

### Opening a Directive

```
:::directive-name key="value" another_key="value"
content here
content continues...
:::
```

- Directive name is alphanumeric + hyphen (e.g. `:::thinking`, `:::tool-call`, `:::tool-result`, `:::stdio`)
- Attributes are quoted key=value pairs separated by spaces
- Directive body is everything between the opening and closing `:::`
- A closing `:::` on a line by itself marks the end of the directive
- **No nesting:** directives do not nest; content between `:::` markers is literal

### Valid Directive Names (v1 Scope)

The following directives are **required** for parsers to recognize:

| Directive | Attributes | Content | Purpose |
|-----------|-----------|---------|---------|
| `:::thinking` | (none) | Plain text | Model's internal reasoning; only in v1 if thinking was enabled |
| `:::tool-call` | `name="<tool_name>"` | JSON | Outbound tool invocation (deferred to v2; v1 does not emit) |
| `:::tool-result` | `name="<tool_name>"` | Plain text or JSON | Tool result returned to model (deferred to v2; v1 does not emit) |
| `:::stdio` | `stream="stdout\|stderr"` | Plain text | Subprocess output from context steps (deferred to v2; v1 does not emit) |

### Version Compatibility Rules

**Unknown directives:** Parsers MUST silently skip directives with unknown names. This enables forward compatibility: if v2 introduces `:::foo-bar`, v1 parsers should not crash.

**Unknown versions:** Parsers MUST error if they encounter a version header other than `ail-log/1` (e.g. `ail-log/2`). Future format changes require a new version number.

Example:
```
ail-log/1
...
:::thinking
reasoning
:::
# Later, v2 adds:
:::token-breakdown in=150 out=75
```

A v1 parser must skip `:::token-breakdown` but must reject `ail-log/2`.

## 4.3 Response Text (Markdown)

Response text — the model's final output for a step — **MUST NOT** be wrapped in a directive. It appears as plain Markdown between directive blocks:

```
ail-log/1

## Turn 1 — `invocation`

:::thinking
Let me understand this request...
:::

The code looks well-structured. Here are my observations:

- Point A
- Point B
```

Markdown features supported:
- Headings (`#`, `##`, `###`, etc.)
- Lists (unordered and ordered)
- Code blocks (`` ``` ``)
- Links, emphasis, strong text
- Blockquotes
- Tables

Response text may span multiple paragraphs and is delimited by the next section header or EOF.

---

## 4.4 Turn Header

Each step in a pipeline run is presented as a **turn**. The turn header identifies the step and its index:

```
## Turn {n} — `{step_id}`
```

- `{n}` is the 1-based sequential turn number (Turn 1, Turn 2, …, Turn N)
- `{step_id}` is the step's identifier in the pipeline (e.g. `invocation`, `review`, `security_scan`)
- The turn number increments for every step that produces output, whether successful, failed, or skipped

Example:
```
## Turn 1 — `invocation`
...
## Turn 2 — `code_review`
...
## Turn 3 — `security_scan`
```

---

## 4.5 Cost Line

Each step's response is followed by a cost line showing token usage and monetary cost:

```
---
_Cost: $0.0015 | 120in / 45out tokens_
```

Format:
- Three dashes (`---`) alone on a line (horizontal rule in Markdown)
- Next line starts with `_Cost: $`
- Cost in USD with four decimal places (e.g. `$0.0015`, `$1.2345`)
- Token breakdown: `{input_tokens}in / {output_tokens}out tokens`
- The entire line is italicized (wrapped in `_` on both sides)

If a step has no cost (context steps, skipped steps, errors):

```
---
_No cost (context step)_
```

Or:

```
---
_No cost (step skipped)_
```

---

## 4.6 Error Callout

If a step fails, an error callout appears after the step response (or in place of it):

```
> [!WARNING]
> **Step failed:** {error_message}
```

Format:
- Markdown blockquote (`>`) with `[!WARNING]` alert box syntax
- Bold step label followed by colon: `**Step failed:** `
- Human-readable error message from the runner or executor

The error callout is the final element of a failed step; no cost line follows.

Example:
```
## Turn 2 — `deploy`

> [!WARNING]
> **Step failed:** Deployment key not found in environment
```

## 4.7 Version 1 Scope and Deferred Directives

### What v1 Emits

In version 1.0, the ail-log formatter emits **only**:

- Turn headers
- Thinking blocks (`:::thinking` directive) — when extended thinking is enabled
- Response text (plain Markdown) — the model's final output
- Cost lines
- Error callouts (if step failed)

### Deferred to v2

The following directives are **specified in this document for forward compatibility** but are **not emitted in v1** because they require a future database schema (`run_events` table):

- `:::tool-call` — requires storing individual tool invocations
- `:::tool-result` — requires storing tool result payloads
- `:::stdio` — requires storing subprocess I/O streams

In v1, the SQLite `steps` table aggregates `thinking` + `response` per turn. Individual tool calls and results are not recorded. When the `run_events` table is added in v2, the formatter will emit these directives.

**Implication for parsers:** v1 parsers will never encounter `:::tool-call`, `:::tool-result`, or `:::stdio` in real v1 output. But they must recognize these directive names (per §4.2) to remain compatible with v2 documents.

## 4.8 Streaming Contract

When invoked with `ail log --follow`, the formatter streams output incrementally:

### Initial Emission

On startup, the version header and all content up to the current point in execution are emitted:

```
ail-log/1

## Turn 1 — `invocation`

:::thinking
...
:::

Response text...

---
_Cost: $0.001 | ..._

## Turn 2 — `analysis`

(In progress...)
```

### Incremental Appends

As new steps complete, their turn blocks are appended to stdout **without re-emitting earlier turns**. Each new turn is a self-contained addition:

```
## Turn 3 — `final_review`

:::thinking
Last thoughts...
:::

The implementation is correct.

---
_Cost: $0.002 | ..._
```

### Streaming Completion

The process exits with code `0` after the final step completes. The last line of output is the cost line of the final step.

On error, the process exits with code `1` and emits an error callout as the last element.

### Parser Implications

Parsers consuming streaming output must:

- Read the version header on line 1
- Buffer input line-by-line
- Accumulate turns as they arrive
- Handle EOF as end-of-stream (no explicit "done" marker)
- Not assume earlier turns are final until end-of-stream is reached (though in practice they do not change)

## 4.9 Complete Worked Example

This example shows one complete pipeline run with three turns, including thinking, response, and costs.

```
ail-log/1

## Turn 1 — `invocation`

:::thinking
The user is asking me to add a fizzbuzz function to a Go program. I should:
1. Ask for the current code structure
2. Propose an implementation
3. Ensure it follows Go conventions
:::

I'll help you add a fizzbuzz function. Let me first see the existing code structure.

What programming language is this for? And do you have an existing file where I should add the function, or should I create a new one?

---
_Cost: $0.0008 | 85in / 42out tokens_

## Turn 2 — `code_review`

:::thinking
The user provided context about the Go program. Now I should:
1. Review the existing code
2. Add a fizzbuzz function that integrates cleanly
3. Explain the implementation
:::

Here's the fizzbuzz implementation I recommend:

```go
func FizzBuzz(n int) string {
    switch {
    case n%15 == 0:
        return "FizzBuzz"
    case n%3 == 0:
        return "Fizz"
    case n%5 == 0:
        return "Buzz"
    default:
        return strconv.Itoa(n)
    }
}
```

This approach uses a switch statement with cases ordered from most specific (divisible by 15) to least specific. It's idiomatic Go and performs well.

---
_Cost: $0.0012 | 156in / 87out tokens_

## Turn 3 — `security_scan`

:::thinking
Now I should perform a security review of the code:
1. Check for injection vulnerabilities
2. Verify error handling
3. Ensure safe Go practices
:::

Security review complete. The fizzbuzz function is safe:

- No external input is processed directly
- No file I/O or network calls
- Conversion to string is safe (strconv.Itoa handles all valid integers)
- The function is deterministic and side-effect free

Approved for production.

---
_Cost: $0.0005 | 98in / 51out tokens_
```

## 4.10 Edge Cases and Recommendations

### Empty Response

If a step produces no text output (e.g., an internal validation step), the turn is still recorded but the response section is empty:

```
## Turn 2 — `validate_config`

(Configuration is valid.)

---
_Cost: $0.0001 | 20in / 5out tokens_
```

### Multi-line Thinking Blocks

Thinking text may span many lines. Each line is preserved verbatim:

```
:::thinking
Line 1 of reasoning
Line 2 of reasoning
Line 3 of reasoning
:::
```

### Code Blocks Within Response

Response Markdown is preserved exactly, including code fences:

```
## Turn 1 — `invocation`

Here's the solution:

```python
def hello():
    print("world")
```

---
_Cost: $0.001 | ..._
```

### Timestamps (Future Extension)

v1 does not include timestamps. If needed in v2, a `recorded_at` directive or metadata header could be added without breaking v1 parsers.

## 4.11 Summary

| Element | Format | Required | Example |
|---------|--------|----------|---------|
| Version header | `ail-log/1` on line 1 | Yes | `ail-log/1` |
| Turn header | `## Turn {n} — \`{step_id}\`` | Yes | `## Turn 1 — \`invocation\`` |
| Thinking directive | `:::thinking` … `:::` | Only if enabled | `:::thinking\nReasoning...\n:::` |
| Response text | Plain Markdown | Optional | `Here's the code...` |
| Cost line | `---\n_Cost: $X.XXXX \| {in}in / {out}out tokens_` | Yes | `_Cost: $0.001 \| 50in / 25out tokens_` |
| Error callout | `> [!WARNING]\n> **Step failed:** {msg}` | Only if failed | `> [!WARNING]\n> **Step failed:** ...` |
